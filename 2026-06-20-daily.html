⭐ 23,100+ TypeScript Trendshift #5

Trendshift #22

stanford-oval/storm

一句话定位：斯坦福出品的 LLM 知识整理系统，给个主题就能自动检索、整理并写出一篇带引用来源的长篇报告。

⭐ 28,900+ Python Trendshift #22
  • 核心思路是「检索 + 多视角提问」：先从多个角度生成问题、联网检索资料，再据此组织大纲、撰写正文，最后产出带引用的完整文章。
  • 被不少人视为比 OpenAI Deep Research 更早的「深度研究」雏形，适合写综述、调研报告、知识科普这类需要溯源的长文场景。
  • 近 2.9 万星、MIT 协议、斯坦福 OVAL 实验室维护，既能当现成研究工具用，也是学习「AI 自动做研究」工程实现的好范本。

今日学一招
CODEX CLI · 实战技巧

用 Codex CLI 安全管理环境变量与密钥：别把 API key 泄漏给模型

让 AI 帮你写代码、跑命令很方便，但项目里往往散落着数据库密码、API key、token 这类敏感信息。一旦它们被读进上下文发给模型，就等于把钥匙交了出去。安全用 Codex CLI 的关键，是让它能把活干完，又始终碰不到真正的密钥明文。

  • 密钥统一放进 .env 文件，并在 .gitignore 与 .codexignore（或等效忽略配置）里排除它，避免 AI 把整份密钥读进上下文。
  • 代码里只引用变量名，例如 os.environ["API_KEY"]，把真实值留在环境变量里；让模型看到「用哪个 key」，而不是 key 本身长什么样。
  • 需要联网或调用工具时，通过 shell 的 export 或运行时注入密钥，而不是写死在代码或贴进对话，命令里也尽量用 $VAR 引用。
  • 用 @ 精准喂上下文时，只引用需要改的源文件，别顺手把 .env、配置后端、私钥目录一起带进去。
  • 万一怀疑某个 key 进过对话或被提交，第一反应是去服务商后台轮换（revoke 并重新生成），而不是寄希望于「应该没人看到」。
