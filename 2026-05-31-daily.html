harry0703 / MoneyPrinterTurbo

🐍 Python ⭐ 80,985+ AI 视频生成 内容创作自动化
输入一个主题，AI 自动配素材、配音、加字幕——一键搞定高清短视频

它是什么、解决什么问题

做短视频最繁琐的不是拍摄，而是后期：找素材、写文案、生成配音、加字幕、剪辑合成……每一步都要开不同的软件，花好几个小时。

MoneyPrinterTurbo 把整条流水线打通了：你只需输入一个主题或关键词（比如"健身减脂的 5 个误区"），它就能自动完成文案生成、视频素材匹配、AI 配音合成、字幕嵌入、视频剪辑，最终输出一段带声音、带字幕的完整短视频——整个过程可以控制在几分钟内。

亮点 / 技术看点

  • 接入 100+ AI 模型：通过 LiteLLM 统一接口，支持 OpenAI、Claude、本地 Ollama 等几乎所有主流大模型——用哪个模型生成文案，随你选，不绑定单一供应商。
  • WebUI + Docker 双入口：提供图形界面，不懂命令行也能操作；Docker 一行命令部署，运维成本极低；还有 Windows 一键安装包，非技术同学直接上手。
  • 素材库灵活接入：内置 Pexels / Pixabay 素材源（免费授权），也可以接入本地素材文件夹，不依赖某一个平台。
  • 多语言配音与字幕：支持中文、英文等多语言配音，字幕可自定义字体、大小、颜色，适配不同平台的发布规范（竖屏/横屏均支持）。
生活类比：以前做一期短视频相当于手工缝一件衬衫——选布料、裁剪、缝制、熨烫每步都要手动完成。MoneyPrinterTurbo 相当于一台全自动服装工厂流水线，你只需说"我要一件蓝色衬衫"，流水线自动完成全部步骤。

学习要点

  • 内容/运营同学：可以用来快速批量生产知识科普类、产品介绍类短视频，验证选题，不必等视频剪辑资源排期。
  • 开发/测试同学：关注它如何把"文本→语音→视频"多个异构 AI 服务串联成一条可靠管道，这是很典型的 AI 工作流编排模式，值得借鉴。
  • 通用视角："一键生成"背后的核心逻辑是把若干个 AI 模型的能力组合起来，每个模型只做一件事，但串联起来就能完成复杂任务。这正是 AI Agent 的基本思路：分工明确，通过编排产生乘数效应。

ruvnet / RuView

🐍 Python ⭐ 34,156+ 硬件 / IoT 隐私感知 边缘 AI
不用摄像头，用 WiFi 信号"看见"人——检测呼吸、心率、跌倒，$9 硬件就能跑

它是什么、解决什么问题

想监测老人是否跌倒、感知会议室里有多少人、或者分析睡眠时的呼吸状态——但又不想在私密空间安装摄像头。有没有办法在保护隐私的同时做到这些？

RuView 给出了一个反直觉的答案：用 WiFi 信号。人体在空间内移动、呼吸，会微妙地改变无线电波的传播路径（这个信号叫 CSI：信道状态信息）。RuView 通过分析这些细微扰动，就能推断出：人在不在？在哪里？呼吸是否规律？有没有突然跌倒？——全程无需摄像头，不存储任何图像，天然保护隐私。

亮点 / 技术看点

  • $9 硬件起步（ESP32）：基于 ESP32 开发板，这是全球最常见的 IoT 芯片之一，在某宝上几十块钱就能买到。整套感知系统的硬件成本极低，人人可复现。
  • CSI 信号分析：WiFi 信号包含振幅和相位两个维度，人体活动引起的微小变化（比如胸腔起伏带来的毫米级位移）都会留下痕迹。边缘 AI 模型在设备本地实时处理，不需要把数据上传到云端。
  • 多种感知能力：人员在场检测（有没有人）、位置定位（人在哪个区域）、生命体征监测（呼吸频率、心率估算）、跌倒检测——这几项能力在传统方案里需要不同种类的传感器，RuView 用一套 WiFi 设备统一实现。
  • 完整 Python 框架：提供数据采集、模型训练、实时推理的完整代码，方便在此基础上自定义场景。
生活类比：想象你在黑暗中听到有人走路——即使看不见，你也能通过脚步声判断"有人，在左边，步伐急促"。RuView 让 WiFi 信号扮演了这双"耳朵"，而 AI 模型负责把信号的细微变化翻译成有意义的感知结论——既不用开灯（不需要摄像头），又能"知道发生了什么"。

学习要点

  • 对硬件/IoT 感兴趣的同学：ESP32 是入门嵌入式/IoT 的好平台，这个项目展示了如何从硬件采集原始信号到 AI 推理的完整链路，是很好的学习案例。
  • 产品/设计同学：这类"非侵入式感知"技术代表了一个重要方向——在不采集生物特征图像的前提下实现智能感知，对隐私保护要求高的场景（养老院、医院、家庭）很有落地价值。
  • 通用视角：现代 AI 的边界已经延伸到感知层。我们平时觉得 AI 只是"回答问题的工具"，但它正在越来越多地接管各类传感器数据的解读工作——这正是"边缘 AI"的核心价值：让智能在数据产生的地方就完成推理，而不是把所有数据都传到云端。

Lum1104 / Understand-Anything

🐍 Python ⭐ 63,481+ AI 编程工具 代码理解 知识图谱
把任意代码库或知识库变成可交互的知识图谱——搜索、提问、探索，一气呵成

它是什么、解决什么问题

接手一个陌生的大型代码仓库，最痛苦的不是写代码，而是"搞不清楚这里的文件之间是什么关系"、"这个函数从哪里来、被谁调用"、"这个模块负责什么"……翻文档、grep 搜索、逐文件读，往往要花好几天才能建立基本认知。

Understand-Anything 用 5 阶段 Agent 流水线自动完成这件事：扫描代码库 → 分析结构 → 分层归类 → 生成导览 → 校验准确性，最终输出一个可交互的知识图谱。你可以在 React 仪表板上模糊搜索任意模块，也可以用自然语言提问，比如"这个项目的认证流程是怎样的"。

亮点 / 技术看点

  • 5 阶段 Agent 流水线：扫描（Scan）→ 分析（Analyze）→ 分层（Hierarchize）→ 导览（Navigate）→ 校验（Validate），每一阶段职责分离，形成可靠的流水线而不是一次性黑盒输出。
  • 输出 JSON 可进 Git：生成的知识图谱数据以 JSON 格式保存，可以 commit 进代码仓库，随代码一起演进——团队所有成员都能受益，而不是只有运行过工具的那个人。
  • React 仪表板可视化：内置前端界面，支持模糊搜索、节点展开、层级浏览和自然语言问答，不需要打开 IDE 就能快速定位知识点。
  • 兼容主流 AI 编程工具：支持 Claude Code、OpenAI Codex、Cursor 等，可以把生成的知识图谱作为上下文输入，让 AI 编程助手更准确地理解你的代码库背景。
生活类比：进入一个新公司，老员工通常要花几周才能搞清楚组织架构和各部门职责。Understand-Anything 就像一个超级全面的"公司百科"：谁负责什么、哪两个部门有协作关系、某个流程从哪里开始到哪里结束——查一下立刻清楚，还能直接问它问题。

学习要点

  • 开发同学：可以在 onboarding 新项目时用它快速建立代码地图，也可以在做 Code Review 前用它快速定位相关模块的上下文；与 Claude Code / Cursor 结合，AI 提建议时会更少"瞎说"。
  • 测试同学：知识图谱能帮助快速识别高耦合模块（牵一发动全身的地方），在回归测试和影响范围分析时非常有用。
  • 通用视角：这个工具背后的核心思路——把隐性知识显性化、把分散信息结构化——不仅适用于代码，也适用于文档库、产品知识库、内部流程库。理解"知识图谱"这个概念，能帮助你在不同场景下辨认同类工具和方案。

给项目写 AGENTS.md——让 Codex 读懂你的代码库

学完本篇，Codex 进入你的项目就像来了个读过"入职手册"的新同事

今天学什么

上期我们学了如何用 config.toml 固化 Codex 的模型和运行参数。但参数只是"行为偏好"，Codex 对你项目本身一无所知——它不知道你的项目用什么框架、测试怎么跑、哪些目录不能动、代码风格是什么要求。

今天要学的 AGENTS.md 正是为了解决这个问题。把它放在项目根目录，Codex 进入这个项目时会自动读取，从此就"懂"这个代码库了。

AGENTS.md 是什么

AGENTS.md 是一个普通的 Markdown 文件，放在项目根目录（或任意子目录）。它的作用是告诉 Codex（以及其他 AI Agent）：

  • 这个项目是做什么的、用什么技术栈
  • 常用命令怎么跑（测试、构建、格式化）
  • 代码风格和命名约定是什么
  • 哪些文件/目录是禁区，不要随便改动
  • 有哪些已知问题或特殊注意事项

你可以把它理解为写给 AI 助手的"入职手册"，写一次，之后每次 Codex 进来都自动读取。

一个典型的 AGENTS.md 示例

# AGENTS.md

## 项目概览
这是一个用 FastAPI + PostgreSQL 构建的内部任务管理 API 服务。
前端是独立的 React 仓库，本仓库只包含后端。

## 技术栈
- Python 3.11 / FastAPI / SQLAlchemy 2.0
- PostgreSQL 15（本地开发用 Docker Compose 启动）
- 测试框架：pytest + httpx

## 常用命令
# 启动本地开发环境
docker compose up -d
uvicorn app.main:app --reload

# 运行所有测试（必须在提 PR 前通过）
pytest tests/ -v

# 代码格式化
ruff format . && ruff check . --fix

## 代码约定
- 所有 API 路由放在 app/routers/ 下，按资源名命名（如 tasks.py）
- 数据库模型放在 app/models/，Pydantic schema 放在 app/schemas/
- 函数和变量名用 snake_case，类名用 PascalCase
- 新增功能必须附带对应的 pytest 测试用例

## 禁区（不要修改）
- alembic/versions/ 目录下的迁移文件，除非明确被要求
- app/core/security.py，涉及鉴权逻辑，改动需专门 Review

## 已知问题
- tests/test_export.py 中有 2 个已知 flaky test，偶尔超时，可以忽略

子目录也可以有自己的 AGENTS.md

如果你的仓库是 monorepo，不同子模块有不同约定，可以在子目录里放各自的 AGENTS.md。Codex 会按"就近原则"读取——在哪个目录里操作，就读取该目录（及其父目录）的 AGENTS.md，离得越近的优先级越高。

实际效果对比：没有 AGENTS.md 时，你可能需要在每次对话开始时花几百字向 Codex 解释"我们项目用 pytest，测试跑 pytest tests/ -v，格式化用 ruff"。有了 AGENTS.md，这些信息 Codex 自动读取，你直接说"帮我给这个函数补测试"就够了——上下文不用重复交代。

写 AGENTS.md 的三条原则

  • 篇幅控制在 400-700 字：太短没用，太长 Codex 会忽略细节，这个区间最有效；
  • 命令要可直接复制执行：写出能直接跑的命令，别写"运行测试"这种模糊描述；
  • 跟着代码演进：换框架、改目录结构时记得同步更新 AGENTS.md，过时的说明比没有说明更危险。
