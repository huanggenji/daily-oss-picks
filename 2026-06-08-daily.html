github/spec-kit

AI 编程 / 规范驱动开发 Python ★ 94.5k

GitHub 官方出品的「规范驱动开发」（Spec-Driven Development）工具箱。它把 vibe coding 的随性流程，换成一条清晰链路：先把想法写成可执行的规范，由 AI 据此生成开发计划，再落地为代码。

不绑定具体模型——主流 AI 编程 Agent（Claude Code、Codex、Cursor、Gemini CLI 等）都能接入，等于给任何 AI 助手套上一层「先想清楚再动手」的工作流。

为什么值得看：当 AI 能一口气写完整个功能时，最大的风险不是写得慢，而是它在解决错误的问题。spec-kit 用规范把人类意图固定下来，让 AI 的产出可审、可控、可复用，是团队规模化用 AI 编程的务实路线。

TauricResearch/TradingAgents

多智能体 / 金融 Python ★ 65.3k

一个用多智能体（Multi-Agents）模拟真实交易公司的 LLM 金融分析框架。它把研究流程拆成不同角色——基本面分析师、情绪分析师、新闻分析师、技术面分析师，再加上看多/看空研究员相互辩论，最后由交易员和风控层汇总决策。

项目附带 arXiv 论文，本质是一套可复现的研究脚手架：你能看到每个角色如何分工、如何交换观点、如何收敛到一个结论。

为什么值得看：它是「多智能体协作」这个抽象概念的一个完整、可读的落地案例。即使你不做量化，也能从中学到如何把一个复杂任务拆成多个有明确职责的 Agent，并让它们通过辩论而非单点判断来提升结论质量。注意：这是研究框架，不构成任何投资建议。

GyulyVGC/sniffnet

系统工具 / 网络监控 Rust ★ 38.4k

一个用 Rust 写的跨平台网络流量监控工具，界面友好、开箱即用。它能实时显示哪些应用正在占用你的网络带宽、连向了哪些地址、用的什么协议，并支持流量统计、地理位置标注和自定义告警。

最新版本主打「按应用区分带宽」，让你一眼看清是哪个进程在偷偷上传下载。

为什么值得看：在 AI 工具满天飞的榜单里，这是一个返璞归真的优秀系统工具——没有云、没有遥测、没有订阅，纯本地运行。对关注隐私、想搞清楚自己机器到底在和谁通信的人来说非常实用，也是学习 Rust 桌面应用与底层网络编程的好范本。

今日学习 · Codex CLI

Codex CLI 的沙箱与审批模式：让 AI 既能干活又不乱来

前几期我们装好了 Codex、写了 AGENTS.md、接了 git 和 MCP。今天聊一个绕不开的核心机制——它执行命令前，到底问不问你？

一、为什么需要沙箱和审批

Codex 不只是写代码，它还会真的去跑命令：装依赖、改文件、执行测试、调用网络。这意味着一句模糊的指令，可能让它删掉不该删的文件，或访问不该访问的地方。沙箱（sandbox）限制它「能动什么」，审批（approval）控制它「动之前要不要先问你」。两者配合，决定了你和 AI 之间的信任边界。

二、三种审批策略

Codex CLI 通常提供由松到紧的几档，用 --ask-for-approval（或交互界面切换）控制：

  • untrusted / 每步都问：最谨慎。任何写文件或执行命令前都暂停，等你确认。适合在陌生或重要的仓库里第一次放手。
  • on-failure / 失败才问：让它在沙箱里自由跑，只有当命令失败或需要更高权限时才来找你。日常开发最常用的平衡点。
  • never / 全自动：完全不打断，配合受限沙箱使用。适合可重跑、可回滚的批量任务，但务必清楚自己开了什么。

三、沙箱权限档位

沙箱常见三档，用 --sandbox 指定：

  • read-only：只能读，不能改文件、不能联网。最安全，适合让它先「看懂代码、给建议」。
  • workspace-write：能改当前工作目录里的文件，但默认不能联网、不能动工作区外的东西。日常推荐。
  • danger-full-access：放开所有限制。只在你完全理解后果时使用。

四、动手试一次

在一个测试项目里，先用最安全的组合让它只读分析：

codex --sandbox read-only \
      "通读这个仓库，告诉我入口文件和主要模块"

确认它理解正确后，再开放写权限、改成失败才问，让它真正动手：

codex --sandbox workspace-write \
      --ask-for-approval on-failure \
      "给 utils.py 里的函数补上类型注解和 docstring"
💡 一句话记住：沙箱管「能动什么」，审批管「动前问不问」。不熟悉的项目从 read-only + 每步都问起步，熟了再逐步放权到 workspace-write + 失败才问。永远不要在重要仓库里一上来就开全自动加完全访问。
