Codex 实战 · 调试

用 Codex CLI 从报错日志反查 Bug：把堆栈直接喂给它

遇到测试失败或运行时报错时，与其自己一行行 grep 堆栈，不如把日志交给 Codex CLI，让它定位根因并给出修复。关键是「给足上下文、让它先解释再动手」。

① 把报错和相关代码一起喂进去 用 @ 引用出错文件，再贴上完整堆栈，Codex 才能把日志行号和源码对上： 
codex "下面这个测试报错，@tests/test_user.py 是用例，
@src/user.py 是实现，帮我先解释报错原因再给修复方案：
[贴上完整 traceback]"
② 让它先「解释」再「改」 明确要求两步走，避免它跳过分析直接乱改：先说清楚是哪一行、为什么触发、影响范围，确认无误后再让它出 patch。这样你能 review 它的判断，而不是盲目接受。
③ 用管道把日志直接灌给它 报错信息长时，可以把命令输出直接喂进来，省去手动复制： 
pytest 2>&1 | codex "这是测试输出，定位失败用例的根因并修复"
④ 修完让它补一条回归测试 根因修掉后，顺手让 Codex 针对这个 bug 写一个最小复现测试，防止同样的问题再回来——「修复 + 回归测试」一次做完。

💡 一句话总结：调 bug 时把完整堆栈 + 相关源码一起给 Codex，要求它先解释根因、再出修复、最后补回归测试，比单纯问「这报错怎么办」靠谱得多。
