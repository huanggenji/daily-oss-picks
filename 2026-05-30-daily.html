OpenBMB / VoxCPM

🐍 Python ⭐ 84,700+ AI 语音合成 今日 Trendshift #1
无需分词就能开口说话——VoxCPM2，开源界最懂声音的大模型

它是什么、解决什么问题

传统的 AI 语音合成（TTS，Text-to-Speech）系统需要先把文字切成一个个「声音 token」的碎片，再逐步拼出语音——就像用乐高积木拼出一段旋律，拼的步骤越多，出错和失真的机会越大，声音也越机械。

VoxCPM2（由清华大学 OpenBMB 团队开源）另辟蹊径：完全跳过分词这一步，采用「扩散自回归架构」（Diffusion Autoregressive），直接从文字生成连续的语音波形。结果是什么？听起来自然得多，还能随意捏造声音——你甚至可以用一句话描述你想要的声音风格，模型就给你生成出来。

亮点 / 技术看点

  • 无分词架构（Tokenizer-Free）：传统 TTS 先把文字编码成离散 token（相当于把歌词切成音节碎片），再一块块还原成声音，层层转换损耗自然度。VoxCPM2 直接端到端生成，省去中间层，类似"直接从乐谱演奏"而非"先把乐谱转成中间格式再演奏"。
  • Voice Design（声音设计）：不需要真人录音，只需用自然语言描述——比如"年轻女性，温柔，带点磁性"——模型就能创造出从未存在的声音。产品经理不再需要录制大量样本音频。
  • 30 种语言 + 零样本克隆：支持英语、中文、日语等 30 种语言。用 3 秒参考音频就能克隆一个说话风格，RTF（实时率）约 0.13，在 RTX 4090 上可实时推理。
  • 完全开源，Apache-2.0：模型权重和代码都开放，支持 LoRA 微调，门槛极低。
生活类比：以前的 TTS 像是"把钢琴曲先翻译成简谱，再翻译成五线谱，最后找人演奏"——多次转换让情感流失。VoxCPM2 像是直接让作曲家一次性完成演奏，没有中间翻译，情感更完整。

学习要点

  • 开发同学：VoxCPM2 提供 Python SDK，可以接入产品的语音播报模块；参考 Nano-vLLM 配合实现轻量部署。
  • 产品/测试同学：Voice Design 功能意味着可以快速原型不同风格的语音体验，不用找配音演员录制大量 demo 就能测试 UX；克隆功能在做 A/B 测试时也很方便。
  • 运维同学：关注其 RTF ~0.13 指标——意味着生成 1 秒音频只需约 0.13 秒 GPU 时间，对在线服务的资源规划有参考价值。
  • 通用视角：开源 TTS 正在突破"听起来像机器人"的瓶颈，未来内部系统的语音播报、通知朗读、无障碍功能都将受益于此类技术。

microsoft / markitdown

🐍 Python ⭐ 51,000+ (Trendshift #3) 开发效率 / 文档处理
把任何文件"喂给" AI 之前，先交给 MarkItDown 翻译成 Markdown

它是什么、解决什么问题

当你想用 AI 分析一份 PDF 报告、一个 Excel 表格或一张包含文字的图片时，最头疼的一步往往不是问问题，而是把内容提取出来——PDF 有乱码、Word 格式混乱、图片里的表格根本无法复制粘贴。

微软开源的 MarkItDown 就是专门解决这一问题的"万能翻译器"：它能把 PDF、Word（.docx）、Excel（.xlsx）、PowerPoint（.pptx）、HTML、图片（含 OCR）、音频（含语音转录）等各种格式，统统转成干净、结构化的 Markdown 文本。

换句话说，MarkItDown 是你把文件投喂给 AI 模型之前的「前处理管道」——它不分析内容，只负责把内容规范化地提取出来，让 LLM 能直接高效读取。

亮点 / 技术看点

  • 格式覆盖全面：几乎覆盖办公场景所有格式——PDF/Word/Excel/PPT/HTML/CSV/ZIP/EPUB/图片/音频，一个工具通吃，不用装 10 个不同的转换库。
  • RAG 友好：输出的 Markdown 保留了标题层级、表格结构、链接等语义信息，比纯文本更适合向量化存入知识库做 RAG（检索增强生成）。
  • 微软 AutoGen 同源：MarkItDown 最初是 AutoGen 多智能体框架内部的数据预处理模块，代码质量经过大规模 AI Agent 工程实战打磨。
  • CLI + Python API 双用法：既可以命令行一键转换，也能在代码里直接调用，非常灵活。
生活类比：MarkItDown 就像餐厅里的"备餐间"——厨师（LLM）只管烹饪，而 MarkItDown 负责把各种食材（PDF、Excel、图片）洗净切好，以统一规格摆上案板，大大提升厨师的效率。

学习要点

  • 开发/运维：在构建内部知识库时，可以用 MarkItDown 批量处理已有 Word/PDF 文档，转出 Markdown 后再向量化；也适合接入文件上传流程，自动预处理后存库。安装一行命令：pip install markitdown。
  • 产品/测试：当你需要把需求文档、测试报告、PRD 投喂给 AI 助手分析时，先用 MarkItDown 转换一下，可以显著提升 AI 的理解准确率，减少格式乱码导致的误解。
  • 通用视角：为什么 Markdown 是 LLM 友好的格式？因为它用简单的符号表达了层次结构，模型能直接理解"## 标题"是章节开头，而 PDF 的富文本格式对模型来说是一堆噪音。理解这一点，就理解了 RAG 工程的核心挑战之一。

unclecode / crawl4ai

🐍 Python ⭐ 20,400+ 数据 / 网页爬虫 / LLM 工程 Trendshift #9
专门为 LLM 而生的开源爬虫——抓回来的数据 AI 能直接读懂

它是什么、解决什么问题

普通的网页爬虫抓到的是 HTML 原始代码，里面混杂着导航栏、广告、脚本等大量与正文无关的噪声。如果你直接把这些内容丢给 LLM，它要么读不完（超出上下文窗口），要么被噪声干扰，理解质量很差。

Crawl4AI 专门针对这个痛点设计：它不仅抓取网页，还会自动清洗、结构化，输出 LLM 友好格式（通常是 Markdown 或 JSON）。支持 JavaScript 渲染的动态页面、截图、批量并发爬取，还内置了抽取策略——你可以用自然语言告诉它"只要正文和作者"，它会帮你过滤掉其余的噪音。

亮点 / 技术看点

  • LLM 抽取策略（Extraction Strategy）：内置基于 LLM 的智能抽取，用一段 prompt 定义你要的字段，框架自动从页面里提取结构化数据，省去大量手写 CSS 选择器的工作。
  • 异步并发，速度极快：基于 AsyncPlaywright 和 Python asyncio，单机即可并发爬取大量页面，速度远超同步爬虫。
  • 智能内容过滤：自动去除导航、广告、cookie 提示等无关元素，直接保留正文——极大减少输出内容的 Token 数，节省 LLM 调用成本。
  • RAG 管道天然对接：输出的 Markdown + 元数据格式，可以无缝接入向量数据库（如 Chroma、pgvector），快速构建基于网页内容的知识库。
工作场景类比：普通爬虫像派一个实习生去复印资料——他把整本杂志（包括封面、广告、索引）都复印回来了。Crawl4AI 像派一个有经验的研究员，他只把你指定的文章正文抄给你，还帮你标好了章节。

学习要点

  • 开发/数据：快速构建内部知识库时，Crawl4AI 可以自动化抓取行业资讯、竞品网站、技术文档，经清洗后直接向量化入库，几十行代码搭出一个 RAG 知识源。
  • 测试/运维：也可以用于监控竞品页面内容变化——定期爬取关键页面，对比差异，自动发出告警。
  • 产品：理解"为什么爬虫抓来的内容不能直接用"是构建 AI 产品时绕不开的基础认知。内容质量直接影响 AI 回答质量，Crawl4AI 这类工具解决的是数据管道的"最后一公里"问题。
  • 通用：Crawl4AI 开源免费，和 MarkItDown 联合使用（先爬取、再二次清洗）是构建 LLM 知识库的常见组合拳。

Codex CLI 配置文件与模型切换

学完本篇，你能自定义 Codex 的默认行为，不用每次都敲一堆参数

今天学什么

上期我们把 Codex CLI 装好了并跑通了第一个命令。你可能发现每次用都要手动指定模型（--model）、设置审批模式（--approval-mode），非常繁琐。今天学的就是怎么把这些偏好固化到配置文件，以及如何在不同项目里切换模型——让 Codex 更懂你的使用习惯。

配置文件在哪里

Codex CLI 的全局配置文件在你的用户主目录下：

# macOS / Linux
~/.codex/config.toml

# Windows
%USERPROFILE%\.codex\config.toml

如果没有这个文件，手动创建它即可。下面是一个典型的配置示例：

# ~/.codex/config.toml

# 默认模型（不填则用 codex-mini-latest）
model = "o4-mini"

# 审批模式：suggest（建议）/ auto-edit（自动修改文件）/ full-auto（完全自动含命令执行）
approval_mode = "suggest"

# 是否启用彩色输出
color = true

# 每次任务最大 token 预算（可控制成本）
max_tokens = 8000

临时切换模型

全局配置用 o4-mini，但这次任务比较复杂，想用更强的 o3，只需在命令行临时覆盖：

# 临时使用更强的模型（只对这次任务生效）
codex --model o3 "帮我重构这个函数，确保有单元测试"

# 查看当前支持哪些模型
codex models

项目级配置（.codex/config.toml）

不同项目可能需要不同的配置——比如一个 Python 后端项目适合 auto-edit 模式，而一个生产系统只应该用 suggest 模式避免误改文件。在项目根目录创建 .codex/config.toml，它会覆盖全局配置： 

# my-project/.codex/config.toml
# 这个项目用更保守的 suggest 模式
model = "codex-mini-latest"
approval_mode = "suggest"

优先级规则：命令行参数 > 项目配置 > 全局配置 > 默认值。这个"就近原则"和大多数工具一样，方便记忆。

一个实际场景

假设你在三个项目之间切换：个人脚本（不在乎成本，用 o3）、公司业务代码（只建议不自动执行，用 suggest）、快速原型（全自动，用 full-auto + mini 省钱）。

只需在每个项目目录放一个 .codex/config.toml，Codex 进入哪个目录就自动用那个配置——就像 .nvmrc 帮你管理 Node 版本一样，无需手动切换。
