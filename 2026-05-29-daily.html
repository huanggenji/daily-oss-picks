microsoft / markitdown

把任意文件变成 LLM 可消化的 Markdown —— RAG 管道的官方标准预处理器

日期 2026-05-29 仓库 github.com/microsoft/markitdown 语言 Python ⭐ 总 Stars ~91,000+ 热度 Trendshift 今日日榜 Top 4，total stars 满足门槛

一句话定位

MarkItDown 是 Microsoft 开源的多格式文档转 Markdown 工具，起源于 AutoGen 项目内部，专为 RAG 管道和 LLM 前处理而生。它把 PDF、Word、Excel、PowerPoint、HTML、图片（含 OCR）、音频、RSS、YouTube 字幕等 15+ 种格式，统一输出成结构化的 Markdown 字符串，让后续的向量化与检索步骤有稳定、干净的输入。

它解决了什么问题

在大量企业 RAG 场景中，文档格式是最大的"脏活"：PDF 有多列乱序、Word 有样式污染、Excel 有合并单元格、PowerPoint 有图文混排。把这些文档喂给 LLM 之前，开发者往往要为每种格式单独写解析器，然后还要手动处理编码、图片 OCR、表格对齐等杂活。

MarkItDown 把这一套全部标准化：

  • 统一出口：无论输入什么格式，输出永远是 Markdown 字符串，上游管道无需分支处理。
  • 全内存操作：所有转换在内存中完成，无临时文件，适合流式处理和容器化部署。
  • 可插拔扩展：内置 Azure Document Intelligence（高精度 OCR + 结构提取）和 LLM 图像描述（将截图自动配文字）两条增强链路，也支持第三方插件注册新格式。

核心架构（架构师视角）

1. 插件注册 + 策略模式的转换器体系

整个系统的核心是一个 DocumentConverter 抽象基类，每种文件格式都有对应的子类实现（PdfConverter、DocxConverter、XlsxConverter……）。运行时通过一个优先级队列式的注册表（converter_registry）按格式匹配转换器，MIME type 与扩展名双重识别。

# 简化示意：注册表工作原理
registry = ConverterRegistry()
registry.register(PdfConverter(), priority=10)
registry.register(DocxConverter(), priority=10)

result = registry.convert("report.pdf")  # 自动路由到 PdfConverter
架构师对照：这是教科书级的策略模式（Strategy Pattern）+ 责任链（Chain of Responsibility）组合——注册表按优先级遍历，第一个 accepts() 返回 True 的转换器接手。对 Java 开发者来说，等价于 Spring 里的 HttpMessageConverter 列表：框架根据 MediaType 自动选转换器，业务代码不感知具体实现。

2. 三层转换链路

层级能力适用场景
基础层纯本地转换，无外部依赖，速度最快CI/CD 内、离线环境
增强层接入 Azure Document Intelligence，精度最高扫描件 PDF、复杂表格
AI 层调用 LLM 为图片生成文字描述截图、图表、手写稿

三层可以独立开启或组合，用环境变量或代码参数控制，不需要改核心代码。

3. 插件体系与 Entry Point 机制

第三方开发者可以通过 Python 的 entry_points 机制注册新格式转换器，用户只需 pip install markitdown-plugin-xxx 并在实例化时传入 enable_plugins=True，插件就会自动发现并注册，对核心代码零侵入。

工程价值：这套插件机制复刻了 Eclipse 插件体系、Pytest 插件体系的核心思路——通过平台标准（Entry Points）实现"开放封闭"，框架稳定，生态扩展。对 Java 开发者，这等同于 SPI（Service Provider Interface）：只要在 META-INF/services/ 注册，框架就能自动发现实现类。

4. 与 AutoGen / RAG 管道的集成

MarkItDown 从 AutoGen 项目脱胎，天然与 Microsoft 的多 Agent 框架深度集成。在 RAG 流水线里，它充当"数据摄入层"的统一接口：任意格式文档进来，Markdown 字符串出去，下游的向量化（Embedding）、分块（Chunking）、检索（Retrieval）步骤可以完全不关心原始格式。

适用场景

  • 企业知识库 RAG 系统的文档预处理层（替换 N 个散装解析库）
  • AI 助手需要"理解"用户上传的任意附件时（Word 报告、Excel 数据、PDF 合同）
  • 内容分析流水线：批量提取文档结构用于分类、摘要、问答
  • 需要对接 AutoGen 多 Agent 框架的项目

给你的学习要点

MarkItDown 是一个"接口设计比算法更重要"的典型案例。它的核心贡献不是某个解析算法，而是一个稳定的抽象边界：无论上游是什么格式，下游永远收到 Markdown。这正是 Java 体系里"面向接口编程"的精髓——HttpMessageConverter、HandlerMethodArgumentResolver、JDK 的 Charset SPI 都是同一个哲学。

🛠 动手建议：用 MarkItDown 搭一个最简 RAG 原型：markitdown("your-doc.pdf") → 文本切块 → text-embedding-3-small → 存向量库 → 检索 → GPT 回答。整个流程 50 行 Python 以内，能让你对"LLM 怎么读文档"建立非常直观的认知。

Lum1104 / Understand-Anything

用多 Agent 流水线把任意代码库变成可交互知识图谱 —— 架构师的"X 光机"

日期 2026-05-29 仓库 github.com/Lum1104/Understand-Anything 语言 Python / TypeScript ⭐ 总 Stars ~38,000+ 热度 Trendshift 今日日榜 Top 20，total stars 满足门槛

一句话定位

Understand-Anything 是一个 Claude Code 插件，它用一条多 Agent 流水线扫描整个代码仓库，提取所有文件、函数、类、依赖关系，构建成一份 knowledge-graph.json，再渲染成一个可交互的 React 可视化仪表板——你可以在图里点击节点、模糊搜索符号、问自然语言问题（"哪里处理了用户鉴权？"），还可以让它自动生成代码库的导览路线。口号是"Graphs that teach > graphs that impress"（能教你东西的图，比好看的图更有价值）。

它解决了什么问题

接手一个 20 万行的老系统是架构师最头疼的场景之一：文档过时、命名混乱、模块边界模糊。传统方法要么靠人肉阅读（太慢），要么靠静态分析工具（输出不可读）。而直接把大量代码丢给 LLM，又会超出 context window，或者让 AI 凭空推断出错误的依赖关系。

Understand-Anything 的解法是：先让 AI 把代码理解成结构化图谱，再用图谱回答问题，绕开了"塞不下"和"幻觉"两个问题。

  • 绕开 context 限制：不是把所有代码一次性喂给 LLM，而是并发地让多个 Agent 各自分析一个文件，最后汇总成图谱。
  • 降低幻觉概率：依赖关系来自静态分析 + AI 提取的双重验证，不依赖 LLM 对代码逻辑的猜测。
  • 增量更新：提交了知识图谱 JSON 到 Git，后续只重新分析变动过的文件，不用每次全跑。

核心架构（架构师视角）

1. 五阶段多 Agent 流水线

整个分析过程由五类 Agent 串行+并行编排完成：

阶段Agent 角色工作内容并发
1project-scanner发现所有文件，识别语言与框架单例
2file-analyzer逐文件提取函数、类、导入关系最多 3 并发
3architecture-analyzer识别分层结构（API/Service/Data/UI）单例
4tour-builder生成可导览的"参观路线"单例
5graph-reviewer最终校验，修正错误关系单例
架构师对照：这个五阶段流水线，本质上是一个有向无环图（DAG）调度器，和 Flink/Spark 的 Stage 划分、Jenkins Pipeline 的 Stage-Step 模型、Airflow 的 DAG 是同构的。"先扫描再分析再校验"这个顺序保证了依赖安全，而阶段 2 的并发则是 Map 阶段——这个经典 MapReduce 结构在 AI Agent 编排里依然奏效。

2. 知识图谱的数据结构

所有分析结果都被持久化到 .understand-anything/knowledge-graph.json，结构是一个标准的图（Nodes + Edges）：

{
  "nodes": [
    {
      "id": "src/service/UserService.java",
      "type": "class",
      "layer": "Service",
      "summary": "处理用户注册、登录、权限校验",
      "patterns": ["singleton", "dependency-injection"]
    }
  ],
  "edges": [
    { "from": "UserController", "to": "UserService", "relation": "calls" },
    { "from": "UserService", "to": "UserRepository", "relation": "depends" }
  ]
}

这份 JSON 建议 commit 进 Git，团队成员 clone 后可跳过全量分析，直接打开仪表板——相当于把"系统理解"像代码一样纳入版本控制。

3. 交互仪表板的前端架构

仪表板是一个 React + React Flow 应用，按架构层（API / Service / Data / UI / Utility）自动分组并着色，支持模糊搜索、语义搜索（问自然语言）、节点详情弹窗（显示函数签名 + AI 生成的摘要）。设计思路更像一个"代码地图"App，而非传统 IDE 的 Call Hierarchy 树。

为什么对 Java 架构师特别有价值：Java 企业项目往往层多、模块多、依赖图复杂。把 Spring Boot 项目丢进 Understand-Anything，几分钟就能得到一张自动标注了 Controller / Service / Repository / Config 分层的可交互图，比手画系统架构图快 10 倍——特别适合做技术评审、新人 Onboarding、或接手遗留系统时的快速摸底。

4. 平台兼容性

目前支持 Claude Code、Codex CLI、Cursor、VS Code + GitHub Copilot、Gemini CLI、OpenCode 等主流 AI 编程工具，作为标准 Skill 安装，无需改动任何项目文件。

适用场景

  • 接手遗留系统时，快速建立对代码库的全局认知
  • 架构评审前，自动生成可分享的系统依赖图
  • 新人 Onboarding：让新成员通过图谱自主探索代码库，而不是靠口口相传
  • 重构计划制定：先看清楚当前的依赖关系，再规划打包边界与模块拆分

给你的学习要点

Understand-Anything 最值得学习的是它的分治 + 汇聚思路——把一个"理解 20 万行代码"这种对单个 LLM 来说无解的任务，拆解成大量"理解一个 200 行文件"的小任务，并发执行后再汇总。这不是 AI 专有的思维，而是所有大规模系统设计的底层逻辑：分而治之、并发处理、聚合结果。

同时，它把"分析结果"序列化成 JSON 持久化，而非每次重新计算，体现了计算与展现分离的工程成熟度——这和 CQRS 里把写模型（Event Log）和读模型（物化视图）分开的思想完全一致。

🛠 动手建议：把自己最熟悉的一个 Spring Boot 项目丢进去跑一遍，然后重点看它识别出的模块分层与依赖关系是否与你脑海中的"心智模型"一致。如果有出入，值得深想一下：是 AI 错了，还是代码里确实有隐式依赖你之前没意识到？

No.3 · Agent 方法论
obra / superpowers

给 AI 编程 Agent 装上工程纪律 —— 一套让 Agent 先想清楚再动手的开发方法论

日期 2026-05-29 仓库 github.com/obra/superpowers 语言 Markdown / Shell ⭐ 总 Stars ~60,600 热度 Trendshift 今日日榜 Top 5，total stars 满足门槛

一句话定位

Superpowers 是一个"AI 编程 Agent 的工程方法论框架"——它不是一个编程库，而是一套可安装到各主流 AI 编程工具（Claude Code、Codex、Cursor、Gemini CLI 等）的结构化技能（Skills）集合，用来约束 Agent 的行为：先澄清需求、再出设计、再写测试、最后才写代码。作者的核心洞察是：AI Agent 最大的问题不是能力不够，而是急于动手、不经设计就开始写代码，导致越改越乱。

它解决了什么问题

用过 AI 编程工具的人都遇到过这个问题：你说"帮我加一个用户注册功能"，Agent 立刻开始改文件，写了一堆代码，表面上看很完整，但接口设计不合理、没有错误处理、测试覆盖为零——而且你没法轻易回退，因为它已经改了七八个文件。

Superpowers 的解决思路是：在 Agent 开始写代码之前，强制插入一个"软件工程流程"：

  • 头脑风暴（brainstorm）：Agent 先从对话中提取真实需求，通过提问澄清模糊点，分块输出设计方案供用户确认。
  • 实现计划（plan）：方案确认后，生成一份清晰到"给一个初级工程师都能执行"的实现计划。
  • TDD 驱动实现：先写测试，再写实现，保证每步都有可验证的检查点。
  • Git Worktree 隔离：在独立分支和工作目录里开发，避免直接污染主线。
  • 子 Agent 协作：复杂任务由主 Agent 分解，派生多个子 Agent 并发执行子任务。

核心架构（架构师视角）

1. Skills 即"可注入的行为约束"

Superpowers 的核心机制是"Skill"——每个 Skill 是一个 SKILL.md 文件，用结构化自然语言定义了"触发条件"（什么时候激活）、"行为指南"（该做什么、不该做什么）和"输出模板"。当 Agent 运行时，会先读取这些 Skill 文件，将它们作为行为约束加载进上下文，相当于给 Agent 注入了"工程师的行为规范"。

架构类比：这非常像 AOP（面向切面编程）——你在不修改业务逻辑（Agent 的核心能力）的情况下，通过切面（Skills）织入横切关注点（代码评审、测试、设计文档）。Spring 的 @Transactional、@Cacheable 就是这种思想：业务方法不知道自己被增强了，但增强行为已经发生。

2. 五个核心 Skill 的编排逻辑

Skill触发时机作用
brainstorming用户要求构建新功能时强制澄清需求，分块输出设计
using-git-worktrees设计确认后创建隔离工作区，避免污染主分支
subagent-driven-development任务复杂度超过单 Agent 时主 Agent 分解任务，派生子 Agent 并发执行
test-driven-development开始编码前先写测试，红绿循环，保证质量
code-review实现完成后自动代码评审，覆盖安全、性能、可维护性

3. "方法论即代码"的哲学

Superpowers 最有意思的地方，是它把"软件工程方法论"本身做成了可版本化、可 fork、可定制的代码资产。你可以 fork 这个仓库，修改某个 Skill 的行为规范，发布自己团队的"工程规范包"，让团队所有人的 AI Agent 都遵循同一套标准。

前瞻思考：这预示着一个新的工程角色——Agent 架构师：不再只是设计系统架构，而是设计"AI Agent 应该如何工作的工作流架构"，制定 Skill 规范、编排约束、质量门禁。这和今天 Java 团队里架构师制定编码规范、Code Review 流程的职责是同构的——只不过执行者从人换成了 Agent。

4. 兼容性与安装方式

Superpowers 支持所有主流 AI 编程工具，安装只需把 Skills 目录拷贝到 Agent 的配置路径，或通过 Playbooks 市场一键安装。不绑定任何特定的 LLM 或编程语言，是一套纯"方法论层"的框架。

适用场景

  • 个人用 AI 编程工具做复杂功能时，避免"写了一堆不可控代码"的混乱
  • 团队统一 AI 辅助编程的工程规范，让所有成员的 Agent 都遵守同一套流程
  • 技术负责人希望在引入 AI 编程的同时保持代码质量和可维护性
  • 研究"Agent 工程方法论"的开发者，理解如何用 Skills 约束 Agent 行为

给你的学习要点

Superpowers 传递的核心洞察是：AI Agent 的问题不是智力不足，而是缺乏工程纪律。这和"聪明的程序员写烂代码"是同一个问题——能力不等于流程，流程才能保障质量。Superpowers 用 Skills 把"好的工程习惯"编码化、可安装化，让 Agent 自动遵守，而不是每次都靠提示词临时约束。

对 Java 架构师的迁移价值：当你给团队引入 AI 编程工具时，不要只想着"让 AI 写代码更快"，更要思考"怎么让 AI 写代码的方式符合团队的工程规范"。Superpowers 给出了一个具体答案——用 Skills 框架把规范固化下来。

🛠 动手建议：读一下 brainstorming/SKILL.md 的原文，看看它是如何用结构化自然语言定义 Agent 行为的。然后思考：如果让你写一份"Java 架构评审 Skill"，你会在里面约定哪些必查项？这个思考过程本身就是在建立你对 Agent 工程化的直觉。

今日三选总结

今天三个项目恰好串联成一条完整的 AI 工程链路：MarkItDown 负责把世界上各种格式的文档"喂给"LLM（数据入口）；Understand-Anything 负责让 LLM 真正理解一个大型代码库的结构与依赖（认知中枢）；Superpowers 则负责约束 LLM Agent 以正确的工程方式使用这些理解成果去写代码（行为纪律）。三者的共同主题是：如何让 AI 在真实工程场景中可信赖、可控制、可持续地工作——这比"让 AI 写更多代码"重要得多，也是 2026 年 AI 工程化的核心议题。
