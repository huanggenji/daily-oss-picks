Ollama：把大模型变成一个本地 HTTP 服务

一个用 Go 写的本地 LLM 运行与服务框架 —— 给架构师的「模型即服务」范本

日期 2026-05-26（周二） 项目 ollama/ollama 语言 Go 定位 本地大模型运行时 仓库 github.com/ollama/ollama

一句话定位

Ollama 让你用一条命令就能在自己的机器上跑起 Llama、Qwen、Gemma、DeepSeek、gpt-oss 等开源大模型，并自动把模型包装成一个标准的本地 HTTP 服务（默认监听 localhost:11434）。对开发者来说，它把"下载权重、配置推理引擎、管理显存、暴露 API"这一整套繁琐工作，收敛成了 ollama run qwen 这样的极简体验——很像当年 Docker 把"跑一个应用"变成 docker run 所做的事。

它解决了什么问题

在它出现之前，本地跑一个开源大模型的门槛相当高：你要自己编译 llama.cpp、手动处理 GGUF 权重格式、根据硬件挑量化等级、写一层 API 包装、还要操心多请求时的显存与并发。Ollama 把这些全部封装掉，并给出三个关键价值：

  • 开箱即用：一条命令完成"拉取模型 → 加载 → 提供服务"，本地隐私数据不出机器。
  • 统一接口：对外暴露一套稳定的 REST API（并兼容 OpenAI API 格式），上层应用无需关心底层用的是哪个推理引擎、哪个模型。
  • 模型分发标准化：用类似 Dockerfile 的 Modelfile 描述模型（基础权重、参数、System Prompt、模板），可复制、可版本化、可分享。

核心架构（架构师视角重点）

Ollama 最值得学习的，恰恰是它的工程架构而非算法。整体是经典的客户端 / 服务端分离：

1. CLI 只是个"瘦客户端"

当你敲 ollama run llama3 时，CLI 并不直接加载模型，而是向本地后台的 Ollama Server 发一个 HTTP 请求，由服务端完成全部重活。这种"命令行只负责发请求、真正的能力沉淀在常驻服务里"的设计，让 CLI、Web UI、第三方应用可以共享同一个后端，是非常干净的解耦。

2. 常驻 Go 服务 + REST API

后端是一个用 Go 写的常驻 HTTP 服务，负责路由请求、管理模型生命周期（加载/卸载/缓存）、调度推理引擎、做流式响应。Go 的轻量协程与原生并发模型，正好适合"多请求、长连接、流式输出 token"这类场景。

对 Java 架构师的镜像：这套结构和我们熟悉的"控制器层 → 服务层 → 资源/引擎层"几乎一一对应。区别在于它把"模型"当成了一种带状态、消耗显存的重资源来管理——加载慢、占用大、需要按需驻留与淘汰，这和连接池、缓存预热、资源隔离的思路相通。

3. 可插拔的双推理引擎

这是 Ollama 架构里最漂亮的一笔。它在服务端抽象出统一的推理接口，背后挂着两个可替换的实现：

引擎实现方式特点
llamaServer通过 CGo 绑定 C++ 的 llama.cpp兼容最广，支持多模态、thinking 模型；量化（如 llama_model_quantize()）也走它
ollamaServer纯 Go 实现的推理引擎更易维护、更可控，目前主要支持纯文本模型

上层只依赖抽象接口，底层换引擎对调用方透明——这就是策略模式 / 依赖倒置在真实大项目里的样子。它既借力了 C++ 生态最成熟的 llama.cpp（用 CGo 桥接），又给出了一条逐步用 Go 自研、摆脱 C 依赖的演进路线，是非常务实的"既要又要"工程取舍。

4. 一次请求的完整链路

ollama run qwen
      │
      ▼  CLI 发 HTTP 请求
  Ollama Server (Go, :11434)
      │  路由 / 加载模型 / 选引擎
      ▼
  Inference Engine (llama.cpp 或 纯 Go)
      │  逐 token 推理
      ▼
  流式返回 token  ──►  CLI / 应用

适用场景

  • 本地 / 私有化 AI 应用：数据敏感、不能上公有云大模型时，用 Ollama 在内网起服务。
  • 开发与原型阶段：本地零成本调试 Prompt、对比不同模型，再无缝切到生产。
  • 给后端应用做 AI 能力底座：因为它兼容 OpenAI API 格式，Java/Spring 应用可以几乎零改造地把请求指向本地 Ollama。
  • 边缘与离线部署：自动探测 GPU/CPU/内存，挑合适的量化版本运行。

给你的学习要点

作为正在补 AI 的 Java 架构师，这个项目最该带走的不是"怎么用"，而是几条可迁移的架构思想：

  • 把复杂能力封装成标准服务：Ollama 用"模型即服务 + 统一 REST API"屏蔽底层复杂度，和我们做中台、做能力网关的思路完全一致。
  • 抽象 + 多实现的可插拔设计：双推理引擎是依赖倒置的范本，思考一下你系统里哪些"重组件"也该这样抽象，以便日后平滑替换。
  • 务实的渐进式演进：先用 CGo 桥接成熟的 C++ 库快速可用，再逐步用 Go 自研替换——对遗留系统改造极有借鉴意义。
  • 把模型当"有状态重资源"来治理：加载、缓存、淘汰、并发隔离，本质和连接池、缓存治理同源，这是 AI 时代后端架构的新课题。

动手建议：装一个 Ollama，用 curl localhost:11434/api/generate 直接调它的 REST 接口看流式返回；再翻一翻仓库里 llm/server.go，看它如何拉起底层引擎进程——半小时就能把"模型即服务"这件事彻底想明白。
