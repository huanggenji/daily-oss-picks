AprilNEA / OpenLogi

🦀 Rust ⭐ 68,310+ 🔥 Trendshift #7 今日 系统工具 开源驱动 隐私优先
用 Rust 写的罗技鼠标官方驱动开源替代品——不注册账号、不联网、零遥测，数据永远在你自己机器上

它是什么、解决什么问题

你买了一只罗技鼠标，想调 DPI、重映射侧键。官方给的方案是安装 Logitech Options+——但这个软件要求你注册账号、持续后台联网，并且会把你的设备信息和使用数据发回罗技服务器，完全关不掉。对许多人来说，这是一个让人不舒服的交换：只是想调个鼠标，却交出了隐私。

OpenLogi 就是这个问题的答案：纯 Rust 实现，通过 HID++（罗技私有硬件通信协议）直接与鼠标对话，支持重映射按键、调节 DPI、SmartShift 滚轮调节，TOML 配置文件管理所有设置；macOS 版本已有菜单栏 App，Linux / Windows 版本正在开发中。全程本地，不需要账号，没有网络请求。

亮点 / 技术看点

  • HID++ 协议直接通信：HID++（Human Interface Device Plus Plus）是罗技自研的扩展通信协议，允许主机软件读写鼠标的高级功能寄存器。OpenLogi 逆向工程实现了这套协议，不依赖任何官方 SDK，完全自主可控——这也是整个项目最硬核的技术部分。
  • Rust 带来的安全保证：直接操作硬件驱动层的软件一旦出现内存 Bug，轻则崩溃、重则导致系统不稳定。Rust 的所有权系统在编译阶段就消灭了 UAF（use-after-free）、空指针这类经典驱动层 Bug，资源占用也极低，开机常驻几乎感受不到。
  • TOML 配置文件：所有配置（哪个按键映射到什么、DPI 是多少）都存在一个人类可读的 TOML 文件里，可以版本控制、团队共享、跨机器同步。与官方 App 把配置锁在云端账号里形成鲜明对比。
  • macOS 菜单栏 App，开箱即用：macOS 用户已经有完整 GUI 界面，菜单栏图标常驻，点开即可调节——不是只有极客才能用的命令行玩具。
生活类比：官方的 Logitech Options+ 就像你买了一台咖啡机，但要调豆子研磨粗细，必须先用手机注册会员，让品牌知道你每天几点喝咖啡，才给你解锁调节按钮。OpenLogi 相当于一个开源改装套件：不需要会员，直接打开后盖，想怎么调就怎么调，数据从不离开你家厨房。

学习要点

  • 使用罗技鼠标的同学：macOS 用户现在就可以试用，替换掉 Options+；Windows/Linux 用户可以 Watch 仓库，等待正式版发布。
  • 对 Rust / 系统编程感兴趣的同学：这是一个非常好的「用 Rust 做实际系统工具」的学习案例——代码量不大，但覆盖了 HID USB 通信、协议解析、配置文件处理、macOS 系统托盘集成等完整链路，适合作为进阶项目研读。
  • 通用视角：「隐私优先的开源替代品」是近两年的明确趋势。越来越多的人开始审视日常软件究竟在后台做什么——像 OpenLogi 这样把「零遥测」作为核心卖点的工具，会越来越受欢迎。

D4Vinci / Scrapling

🐍 Python ⭐ 46,656+ 🔥 Trendshift #12 今日 数据 网页爬虫 反爬
自适应网页爬虫框架——页面改版后自动重新定位元素，CSS 选择器再也不用手动维护

它是什么、解决什么问题

任何做过爬虫的人都有这个噩梦：辛辛苦苦写好一套爬虫，对方网站一改版，所有 CSS 选择器全部失效，脚本哑火，维护工作量和写代码差不多大。这是传统爬虫的核心脆弱性——它依赖页面结构的稳定，而页面结构从来就不稳定。

Scrapling 的核心创新是「自适应」：它会学习目标元素在页面上的多维位置特征（标签路径、相邻元素、文字内容特征等），当页面结构变动后，自动用新的路径重新定位同一个元素——不需要你手动更新任何选择器。在此基础上，它还内置了 Cloudflare Turnstile 等主流反爬机制的绕过能力、并发爬取、代理轮换、多 Session 管理，测试覆盖率达到 92%。

亮点 / 技术看点

  • 自适应元素定位：这是 Scrapling 最独特的能力。它不只存一条 CSS 路径，而是用机器学习思路记录元素的「指纹」——位置、层级关系、周围文字……哪怕 DOM 结构大幅重构，只要目标元素还在页面上，它大概率能重新找到它。
  • 内置反爬绕过：Cloudflare Turnstile 是当前最主流的 Bot 检测机制之一，普通 requests 或者 Playwright 直接被拦截。Scrapling 内置了相应的绕过策略，对于不涉及 JS 渲染的页面，可以在不启动完整浏览器的情况下通过检测，速度远快于 Selenium/Playwright 全量方案。
  • 并发 + 暂停恢复：支持多个任务并发运行；爬取过程可以暂停、持久化状态、下次启动从断点继续——这对需要爬取大量页面的长时间任务至关重要。
  • 代理轮换 + 多 Session：内置代理池集成和 Session 隔离，适合需要模拟多个独立用户身份的爬取场景，开箱即用，无需自行拼装。
生活类比：传统爬虫就像你凭记忆去超市找东西——你记住的是「酸奶在第三排左起第二个格子」。超市一调整陈列，你就找不到了，只能重新记。Scrapling 更像一个认识商品的购物助理：它记住的是酸奶长什么样、旁边通常放什么——不管超市怎么重新摆放，它都能帮你找到酸奶。

学习要点

  • 有数据采集需求的同学：如果你有长期运行的爬虫（监控价格、采集内容、竞品分析），Scrapling 的自适应能力可以大幅降低维护成本，值得迁移试用。
  • 对反爬感兴趣的同学：Scrapling 的源码是学习「主流反爬机制是如何工作的，以及如何合理绕过」的好材料——理解攻防双方的逻辑，是做好数据工程的基础知识。
  • 通用视角：「自适应」是未来软件工具的大方向——不依赖固定规则、能自动应对变化，是从脆弱系统走向鲁棒系统的关键一步。爬虫只是最早出现这个需求的场景之一。

chopratejas / headroom

🐍 Python ⭐ 30,220+ 🔥 Trendshift #14 今日 AI 基础设施 LLM 工程 成本优化
LLM 应用的 Context 压缩层——把喂给 AI 的内容智能压缩 60~95%，费用和速度同步改善

它是什么、解决什么问题

当你让 AI 处理一段日志、一个工具调用的输出、或者一堆 RAG 检索结果时，这些内容往往塞满了 AI 根本不需要的冗余信息：重复的栈帧、样板代码、无关的 JSON 字段……结果就是大量 token 被浪费在「投喂给 AI 的噪音」上，账单膨胀，响应也变慢。

headroom 是一个插在你的应用和 LLM 之间的「压缩层」：它不是粗暴截断，而是智能提取关键内容——错误信息、异常堆栈、与查询相关的代码段、RAG 里真正有用的 chunk——把无关的噪音去掉，压缩比可以达到 60~95%，而 AI 看到的信息质量基本不降。内置 OpenAI 兼容代理模式，只要把 base_url 改成 headroom 的地址，现有代码零改动接入；CacheAligner 模块还能稳定 KV Cache 命中率，进一步降低 API 成本。

亮点 / 技术看点

  • 智能提取而非粗暴截断：headroom 会分析内容的结构和语义，针对不同内容类型用不同策略压缩：日志提取错误行 + 上下文；代码保留函数签名 + 关键逻辑；RAG chunks 按相关性打分过滤——和直接截取前 N 个 token 相比，保留的信息质量差别非常大。
  • OpenAI 兼容代理模式：这是工程集成最聪明的设计。headroom 可以作为一个 API 代理运行，对外暴露完全兼容 OpenAI 格式的接口。你只需要把现有代码里的 base_url 从 OpenAI 的地址改成 headroom 的本地地址，所有压缩逻辑自动生效，其他代码一行不改。
  • CacheAligner：LLM API 通常提供 KV Cache 功能——相同的前缀内容命中缓存后费率大幅降低。但如果每次请求的 prompt 前缀稍有变化，缓存就会失效。CacheAligner 通过标准化和稳定化 prompt 的固定部分，让缓存命中率最大化，和压缩一起叠加使用，节省效果更显著。
  • 多场景覆盖：tool outputs（Agent 调用工具的返回结果）、日志、RAG chunks、代码、图片——常见的高 token 消耗场景都有对应的压缩策略，不是只解决一个点。
生活类比：给 AI 喂原始日志，就像你把一份 200 页的原始会议记录交给秘书，让他提炼成一份摘要给老板汇报。老板真正需要的其实只有 10 页的决策要点，其余 190 页都是过程细节。headroom 就是那个懂业务的秘书：它不是随机扔掉 190 页，而是认真读完，只保留老板真正需要的部分——老板（AI）做出的决策质量没变，但你不用为那 190 页多余的内容付打印费（token 费）了。

学习要点

  • 在用 LLM API 的开发同学：如果你的应用有大量 tool calling、日志分析或 RAG 检索，headroom 可能是当前投入产出比最高的成本优化手段——代理模式接入几乎没有迁移成本。
  • 对 LLM 工程感兴趣的同学：headroom 的内部实现是学习「如何针对不同内容类型设计压缩策略」的好材料。这里涉及的思路（结构化分析 + 相关性评分 + 缓存稳定化）是 LLM 应用工程的核心能力点。
  • 通用视角：「AI 成本」正在变成 AI 落地的第一道门槛。聪明的压缩不是在质量和成本之间妥协——而是让两者同时变好。headroom 代表了一类新兴工具的方向：不是更强的模型，而是更聪明地用现有模型。

Codex CLI × git 集成——读 diff、写 commit message，一行命令搞定

学完本篇，你能让 Codex 自动读懂你的代码改动，生成规范的提交信息，还能挂进 git hook 全自动触发

背景：为什么 commit message 总是写不好

提交代码时最难受的一步不是写代码，而是写 commit message。"fix bug"、"update"、"改了一下"……这些无意义的提交信息在几个月后回溯代码时几乎没有价值。写好 commit message 需要你在完成代码改动之后，重新用语言整理一遍「我改了什么、为什么改」——这既费时间，又容易被跳过。

Codex CLI 和 git 组合起来，可以把这件事变成一行命令：让 Codex 读取你的 diff，自动生成符合 Conventional Commits 规范的提交信息，甚至直接帮你完成 commit。

用法一：先理解改动，再决定要不要提交

先让 Codex 用中文解释这次 diff 做了什么，确认理解无误后再生成 message：

$ git diff HEAD | codex "解释这次 diff 做了什么，用中文简洁说明改动的目的和影响"

↑ 可直接复制使用，把 git diff HEAD 替换成 git diff main 可以对比分支差异

如果只想看 staged（已 git add）的改动：

$ git diff --staged | codex "用一句话总结这次暂存的改动内容"

↑ --staged 等价于 --cached，只分析已经加入暂存区的文件

用法二：直接生成 commit message 并提交

让 Codex 按 Conventional Commits 格式（feat: / fix: / refactor: 前缀）生成提交信息，并直接执行 commit：

$ git diff --staged | codex "根据这次 diff 生成一条符合 Conventional Commits 规范的 commit message，格式为 type(scope): subject，然后执行 git commit -m '<message>'" --auto-edit

↑ --auto-edit 让 Codex 在沙箱中直接执行 shell 命令；去掉此参数则只输出 message 不自动提交

只想要 message 文本、自己决定要不要用：

$ git diff --staged | codex "输出一条 Conventional Commits 格式的 commit message，只输出 message 本身，不要任何解释"
# 复制输出内容，然后：
$ git commit -m "feat(auth): add OAuth2 refresh token rotation"

↑ 两步走更安全：先看 Codex 给出的 message，满意了再手动 commit

进阶：用 git hook 全自动触发

把上面的命令挂进 prepare-commit-msg hook，每次 git commit 时自动调用 Codex 生成初稿，你在编辑器里确认后保存退出即提交：

# 在项目根目录执行，创建 hook 脚本
$ cat > .git/hooks/prepare-commit-msg << 'EOF'
#!/bin/sh
# 只在普通 commit 时触发（不处理 merge / amend）
[ "$2" = "merge" ] || [ "$2" = "commit" ] && exit 0
MSG=$(git diff --staged | codex "输出一条 Conventional Commits 格式的 commit message，只输出 message，不要多余文字" 2>/dev/null)
[ -n "$MSG" ] && echo "$MSG" > "$1"
EOF
$ chmod +x .git/hooks/prepare-commit-msg

↑ 完整可复制，粘贴到终端执行即可；hook 写入后对当前仓库所有后续 commit 生效

安装完成后，每次运行 git commit（不带 -m），编辑器会弹出 Codex 自动生成的 message 草稿，你直接确认或微调后保存退出，提交完成。

小贴士：让生成结果更准确

  • 控制 diff 大小：Codex 一次能处理的上下文有限，一个提交改动 5~10 个文件效果最好；改动太多时先 git add -p 分批暂存，拆成多次小 commit。
  • 在 prompt 里指定语言和格式：如果你的团队要求英文 message，在 prompt 里明确写"用英文输出"；如果有自定义前缀规范，也一起说清楚，Codex 会严格按要求生成。
  • 对比 diff 模式：用 git diff main...HEAD 可以把整个 feature branch 的改动汇总成一个 diff 传给 Codex，让它生成一条概括整个分支工作的 PR 描述，比逐条 commit 更适合写 Pull Request 标题和 body。
